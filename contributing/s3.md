# S3 и деплой инфраструктура проекта Kit Docs

## Обзор архитектуры

Проект использует GitLab CI/CD для автоматического деплоя документации через Helm charts в Kubernetes с nginx proxy и синхронизацией статических файлов в S3.

## Nginx конфигурация

**Файл:** [nginx.conf.template](../nginx.conf.template)

### Ключевые особенности:

- **Прокси на CDN S3**:
  ```nginx
  location ~ ^/(img|video)/(.*)$ {
      set $s3_path "${S3_BUCKET_NAME}/static/$1/$2";
      resolver coredns.kube-system.svc valid=10s;
      proxy_pass https://disk.2gis.com/$s3_path;
  }
  ```
- **Резолвер нужен** для динамического разрешения переменной `${S3_BUCKET_NAME}` и $1, $2. Nginx пытается разрешить все хосты, указанные в директивах proxy_pass, уже при запуске, и не сможет запуститься, если эти хосты недоступны.

## CI/CD Pipeline - GitLab

### S3 синхронизация

**Файл:** [.gitlab/ci/templates/s3-sync.gitlab-ci.yml](../.gitlab/ci/templates/s3-sync.gitlab-ci.yml)

- Использует `s3cmd` с конфигом из `.s3/.s3cfg`
- Синхронизирует `./static/img/` и `./static/video/` с S3 bucket
- Добавляет креды через переменные `AWS_ACCESS_KEY_ID` и `AWS_SECRET_ACCESS_KEY`

### Master vs MR джобы (S3)

**Файлы:**

- [.gitlab/ci/master.gitlab-ci.yml](../.gitlab/ci/master.gitlab-ci.yml) - содержит `s3 sync prod` джобу → деплоит в **продакшн S3 bucket**
- [.gitlab/ci/mr.gitlab-ci.yml](../.gitlab/ci/mr.gitlab-ci.yml) - содержит `s3 sync staging` джобу → деплоит в **staging S3 bucket**

**Разные S3 bucket'ы для разных окружений:**

- **Master**: `S3_BUCKET_PROD`, `AWS_ACCESS_KEY_ID_PROD`, `AWS_SECRET_ACCESS_KEY_PROD`
- **MR**: `S3_BUCKET_STAGING`, `AWS_ACCESS_KEY_ID_STAGING`, `AWS_SECRET_ACCESS_KEY_STAGING`

**Как это работает:**

- При создании MR → запускается `s3 sync staging` → статические файлы попадают в staging bucket
- При мердже в master → запускается `s3 sync prod` → статические файлы попадают в production bucket
- Nginx в разных окружениях использует переменную `S3_BUCKET_NAME`, которая подставляется в прокси URL
- Staging и production имеют свои отдельные S3 креды для изоляции доступа

Все переменные настраиваются в GitLab CI/CD Variables проекта.

## Filestash (веб-интерфейс для S3)

Иногда нужно посмотреть, что реально лежит в S3 bucket'ах, или удалить какие-то файлы вручную. Для этого есть [Filestash](https://filestash.2gis.tech/) - веб-интерфейс для просмотра S3 storage.

Чтобы зайти, нужны ACCESS_KEY и SECRET_KEY из Vault'а. Секреты называются `s3_public_docs_user` для продакшена и `s3_public_docs_user_staging` для staging окружения. Получить их можно в [Vault](https://vault.2gis.tech/ui/vault/secrets).

Это особенно полезно, когда что-то пошло не так с синхронизацией в CI и нужно проверить, попали ли файлы в нужный bucket.

## Как работают медиа-файлы в разных окружениях

Когда добавляешь картинки или видео в документацию, их нужно положить в папки `/static/img/` или `/static/video/` соответственно.

**Локально** - когда запускаешь `npm run start`, Docusaurus сам раздает файлы из папки `static/`. Картинка `static/img/example.png` будет доступна по адресу `http://localhost:3000/img/example.png`.

**В production и staging** все работает по-другому. Там крутится Docker контейнер с nginx, который настроен так, что запросы к `/img/*` и `/video/*` не идут к файлам внутри контейнера, а проксируются на внешний CDN `disk.2gis.com`.

Файлы туда попадают автоматически через CI джобы: когда создаешь MR - файлы синхронизируются в staging S3, когда мерджишь в master - в production S3. Поэтому пользователи получают картинки не из Docker контейнера, а через быстрый CDN.

Если что-то не работает с картинками в staging/production, первым делом проверь логи джоб `s3 sync staging` или `s3 sync prod` в GitLab CI.
