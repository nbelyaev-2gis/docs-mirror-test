## CI/CD

### Сруктура

> Текст ниже носит рекомендательный характер.

Любой файл `.gitlab/ci/{name}.gitlab-ci.yml` представляет собой пайплайн для конкретной задачи. Такой файл содержит внутри (без неявных импортов из других модулей) все джобы, которые должны присутствовать в пайплайне.

Во избежание дублирования кода заводятся шаблоны в директории `.gitlab/ci/templates/`.

Пример одинаковых по сути джоб в разных пайплайнах.

```yml
# mr.gitlab-ci.yml
storybook build:
  extends: [.storybook build]
  needs: [deps build]

# master.gitlab-ci.yml
storybook build:
  extends: [.build_storybook]
  needs: [deps build]
```

Файлы-пайплайны добавляются в пайплайн через директиву `include` в `.gitlab-ci.yml`.

### Что писать внутри шаблона, а что в файле-пайплайне?

#### Файл-пайплайн

- Порядок/зависимости выполнения джоб.
- Условия выполнения джоб.
- Директивы, которые относятся к конкретной джобе в конкретном пайплайне.

Цель — описать что и как выполняется в джобе.
Примеры директив джобы, которые следует писать в файле-пайплайне:

- rules:
- only: / except:
- needs:
- dependencies:
- when:
- allow_failure:
- stage:

#### Шаблон

- Стараемся писать функциональные штуки и их зависимости. Например, нам надо описать шаблон helm lint

  ```yml
  .helm_test:
    script:
      - helm lint ./deploy/storybook

    # Для такого скрипта точно нужен образ, где этот скрипт запускается.
    image:
      name: alpine/helm:3.8.0
      entrypoint: ['']
  ```

Цель — описать когда, где и зачем выполняется джоба.
Примеры директив джобы, которые следует хранить в файле-шаблоне:

- script:
- image:
- variables:
- artifacts:
- interruptible: и retry:
- extends:

### Зачем?

Чтобы генерация джоб внутри пайплайна стала прозрачной. Чтобы не надо было исследовать зависимости появления джобы в пайплайне, просматривая много файлов.
